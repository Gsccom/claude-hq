# Review Criteria — Doubt-Driven

## 사용법
코드 작성 후 clean context(새 눈)로 아래 질문에 답하라.
하나라도 "아니오"면 수정 후 재검토.

---

## 1. 보안 검토
- [ ] API 키가 클라이언트 코드에 노출되지 않는가?
- [ ] 환경변수는 서버사이드에서만 접근하는가?
- [ ] 사용자 입력에 XSS 방어가 되어있는가?

## 2. 에러 처리 검토
- [ ] 모든 async 함수에 try/catch가 있는가?
- [ ] API 실패 시 사용자에게 명확한 메시지가 보이는가?
- [ ] Supabase 오프라인 시 앱이 죽지 않는가?
- [ ] 스트리밍 중 컴포넌트 unmount 시 cleanup되는가?

## 3. 성능 검토
- [ ] 불필요한 re-render가 없는가?
- [ ] 이미지/폰트 최적화가 되어있는가?
- [ ] 데이터 캐싱이 적용되어있는가?
- [ ] 큰 데이터셋에서 가상 스크롤이 필요한가?

## 4. 코드 품질 검토
- [ ] 컴포넌트가 단일 책임 원칙을 따르는가?
- [ ] 중복 코드가 없는가?
- [ ] 변수명이 명확한가?
- [ ] TypeScript 타입이 정확한가?

## 5. YeshuaWorld 규칙 검토
- [ ] 설교 파일 삭제 코드가 없는가?
- [ ] KJV 성경 인용이 정확한가?
- [ ] NAS 경로가 올바른가? (/volume1/...)
- [ ] Supabase URL이 환경변수인가?

## Doubt-Driven 질문 5가지
1. "이 코드가 실패할 수 있는 가장 황당한 경우는?"
2. "아버지가 처음 사용할 때 막힐 부분은?"
3. "6개월 후 내가 이 코드를 볼 때 이해할 수 있는가?"
4. "이 기능을 더 단순하게 만들 수 있는가?"
5. "만약 이 코드가 프로덕션에서 실패한다면 어떻게 복구하는가?"
